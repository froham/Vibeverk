### 6f — Motta e-post (inbound) ⏳ *designa, sett på vent*
Per no finst det INGEN inbound-e-post — `send-reply` (Resend) er reint utgåande. Design ferdigstilt med arkitekt-agent 2026-07-01 (to rundar), avklart med brukar. Gjeld berre `crmFull`-kundar — andre bruker Outlook, ingen endring for dei. **Sett på vent av brukar same dag** ("Vi avventer litt, det blir veldig edgy-CRM-messig").

**Mottak:** Eitt subdomene (t.d. `mail.vibeverk.no`) som catch-all hos Resend (stadfesta via Resend sin eigen dokumentasjon — dei støttar dette nativt, ingen tredjepartsleverandør trengst). MX-records på subdomene for å ikkje forstyrre eksisterande e-post.

**Flyt:** Resend-webhook (`email.received`, kun metadata + signatur) → verifiser svix-signatur → hent full e-post (headers + body) via Resend sitt Receiving API med `message_id` → parse `In-Reply-To`/`References` → slå opp i ny `outbound_emails`-tabell (`lead_id ↔ resend_message_id`, må populerast av `send-reply` som i dag berre returnerer denne IDen utan å lagre ho) → matcha tråd får ny rad i ny `lead_messages`-tabell + `status:"ny"` på leaden. Matching skjer på tråd, ikkje mottaksadresse — fungerer uansett kva "Svar-til"-adresse admin brukte for den enkelte samtalen.

**Ingen match → automatisk ny Kontakt-lead + ny CRM-kunde** (brukar-krav, ikkje manuell kø): ny lead via same skjema som `addLead()` (`core.js:345`), ny `source:"epost-innkommande"`. CRM-kunde oppretta/gjenbrukt via delt `findOrCreateCustomerByEmail()` (ny, gjenbruk dedup-mønster frå `module-crm.js` linje 236/670). Krev nytt `customerId`-felt på lead (additivt, bakoverkompatibelt).

**Kritisk byggbarheits-punkt (ikkje løyst enno):** `leads` og CRM-kundar er i dag JSON-blobbar i éin `store`-nøkkel (localStorage-speglingsmønster), ikkje normaliserte Supabase-rader. Ein Edge Function kan IKKJE skrive rett til denne blobben (race mot admin som redigerer samtidig i UI). Inbound-oppretta leads/kundar må først leggjast som normaliserte rader (nye tabellar), som klienten so pollar inn og fletter inn i `leads`/CRM-lista på neste synk — same mønster som chat allereie bruker. Dette må spesifiserast konkret før bygging.

**Andre krav:**
- Spam/misbruk: sjekk SPF/DKIM/DMARC i dei fulle headera (Resend eksponerer ikkje spam-score i webhooken), rate-limit auto-oppretting per avsendar-domene/døgn, saner emne/brødtekst før lagring (XSS — vert vist i CRM-tidslinje)
- Må gjennom **Security Auditor** (nytt offentleg endepunkt, automatisk oppretting av kundedata frå ukjende/uverifiserte avsendarar) og **Privacy Advisor** (personopplysningar i e-postinnhald) før bygging — sjå CLAUDE.md AI-agent-arbeidsflyt
- `intranet/module-settings.js` viser i mellomtida eksplisitt «Mottak av e-post er ikkje støtta enno»

## Steg 10 — AI-native Chat 🤖
*Mogleg etter Supabase er på plass (steg 6)*

### Kva det er
Ein AI-assistent bygd inn i den native chat-løysinga. Kundar får automatiske,
relevante svar basert på verksemda sitt eige innhald. Menneske kan gripe inn.

### Arkitektur
```
Kunde skriv → Supabase Edge Function → Claude API → Svar til kunde
                      ↑
        Kunnskapsbase + FAQ + CRM + Booking (RAG)
```

### Funksjonar
- **RAG (Retrieval-Augmented Generation)** — AI-en finn relevante KB-artiklar
  via pgvector (innebygd i Supabase) og svarar basert på desse
- **CRM-kontekst** — AI-en veit kven kunden er og kva historikk dei har
- **Booking-integrasjon** — kan foreslå ledige tider direkte i samtalen
- **Hybrid-modus** — AI svarar fyrst, admin kan overta når som helst
- **Usikker-flagging** — AI markerer saker den ikkje er sikker på
  → admin prioriterer desse

### Forretningsmodell
```
Basis:   Manuell chat (allereie bygd ✅)
Pro:     AI-assistent basert på KB og FAQ
Premium: Full AI-agent med CRM + Booking-integrasjon
```

### Kvifor Vibeverk har eit fortrinn
Alle datakjeldene ligg på same stad: KB, CRM, FAQ, Booking.
Dei fleste konkurrentar har ikkje denne integrasjonen.
