## Implemented and verified

- **Three delivery surfaces**: public website (`/`), Workspace/intranet (`/intranet/`), Vibeverk Operator Console (`/console/`) — see `docs/architecture/system-overview.md`.
- **Public site modules**: booking, chat (widget + admin), CRM (light), FAQ, mediabank, quote/tilbud, references, scroll banner, contact/kontakt leads, user administration.
- **Workspace modules**: dashboard, announcements, tasks, notes, links, CRM, knowledge base, org drift, settings, media bank (internal), booking/quote/contact admin views. Note: several "shared" modules (`module-chat.js`, `module-crm.js`) are a single root-level file that dual-registers for both Web-admin (`App.registerModule`) and Workspace (`window.Intranet.registerModule`) — not two separate per-surface implementations. See "Known limitations" for a discovered dead-code duplicate (`intranet/module-crm.js`).
- **Roles**: `admin`/`editor`/`member` only — the database CHECK constraint on `users.role` does not permit any other value. An earlier `owner` role was removed (commit `2f8a92b`/`hotfix_roles.sql`); lingering references across code, tests and docs were cleaned up 2026-07-01, see `docs/decisions/ADR-0006-remove-owner-role-references.md`. Only `admin`/`editor` can assign a task to another user (`intranet/module-tasks.js`); `member` can create tasks for themselves and fully edit tasks they created themselves. Tasks assigned to `member` by someone else are **fully read-only** (view-only detail modal, not even status-changeable) — corrected across three rounds 2026-07-02 after user clarification (first "can't create at all" → "can create for self" → "can fully edit own" → finally "assigned-by-others is fully read-only, not status-only"). Two of the three coordinated RLS/trigger changes this required were run against production 2026-07-02; the third (`hotfix_tasks_readonly_for_assigned_2026-07-02.sql`) is prepared but not yet run — see "Security findings status" below. As of 2026-07-02, member/editor UI restrictions in `intranet/module-dashboard.js`, `module-tasks.js`, `module-mediabank-internal.js` and `module-orgdrift.js` are backed by handler-level checks (not just hidden buttons) — see `docs/architecture/roles-and-tenants.md` for the full matrix, including the org-drift admin-only-write decision. - `config.js → admin.password` is still the placeholder value `"test"`. As of 2026-07-01 this is no longer reachable in production (see Auth note above), but should still be changed before any real production customer use — it remains the fallback for local/test environments without Supabase, and a placeholder value there invites confusion.
- Web-admin password authentication, when Supabase is not configured (test/local environments only), is enforced client-side only — not a production security boundary. See `docs/security/security-baseline.md`.
- `storageKey: "nordpunkt"` cannot be renamed without a full atomic data migration — existing Supabase rows and localStorage entries are keyed to it.
- `AGENTS.md` and `CLAUDE.md` are maintained as two separate, near-duplicate files (one per AI harness) rather than a single generated source — keeping them in sync is a manual, currently unenforced process.
- **`intranet/module-crm.js` is dead code — file itself not yet deleted, pending an explicit decision.** `intranet/index.html` loads the root `module-crm.js` instead (via `../module-crm.js`), which dual-registers for both Web-admin and Workspace — the separate `intranet/module-crm.js` file is never loaded by any real page. Discovered 2026-07-01. As of 2026-07-02, `test-intranet.js` now loads and tests the active root `module-crm.js` (previously it hardcoded evaluation of the dead `intranet/module-crm.js`, so CRM tests exercised a file that never ran in production) — the *test-coverage* gap is closed, but the dead file itself is intentionally left in place, since deleting it is a separate decision not requested in this pass. `intranet/module-crm.js` should not be trusted as a reference for current CRM behavior — read root `module-crm.js`.
- `is_admin_or_owner()` SQL function name is historical (only checks `role = 'admin'`) — not renamed since it's referenced by many RLS policies; would need a coordinated migration. See `docs/decisions/ADR-0006-remove-owner-role-references.md`.
- **`.help-icon__pop` (the `C.helpIcon()` tooltip popover) has no viewport/ancestor-edge clamping.** Flagged by UX review 2026-07-02: the popover is `position:absolute; left:50%; transform:translateX(-50%); width:230px`, centered on the "?" button with no awareness of the viewport edge or of `overflow:hidden` ancestors (e.g. the rounded-corner editor card wrapper in `intranet/module-announcements.js`'s "Ny/rediger Aktuelt-sak" editor). On narrow mobile widths, a "?" button positioned left-of-center can have roughly half its popover clipped by the card's `overflow:hidden`. This CSS/HTML pattern is shared identically across `index.html`, `admin/index.html` and `intranet/index.html` (not something introduced or made worse by the 2026-07-02 Workspace tooltip fix — that fix corrected a *different*, JS-binding-level bug, see 0.9.0 changelog) — fixing the clipping would mean touching a shared, already-in-production CSS pattern across all three surfaces, out of scope for the narrow fix requested this round. Left for a future pass; smallest likely fix is either removing `overflow:hidden` from the editor-card wrapper or making `.help-icon__pop`'s position edge-aware (similar to the clamping added to the reply-snippet dropdown's `positionDd()` in `core.js` the same round).

### Fixed in production, confirmed run by user 2026-07-01
`supabase/hotfix_security_audit_2026-07-01.sql` was run by the user in the Supabase Dashboard SQL Editor against the live project (`clzczbyklgdtdhgjphup`) — confirmed success ("Success. No rows returned"). Same fixes also folded into `supabase/migration.sql` for future fresh customer projects. Noted here as user-confirmed rather than repo-verified, since SQL execution against Supabase happens outside version control (same caveat as `hotfix_visitor_rpcs.sql` below).
- **HIGH — self-escalation to admin via `users.role`.** `users_self_update`'s `USING`/`WITH CHECK` only verified row ownership, not that `role` stayed unchanged (RLS can't restrict individual columns) — any member could PATCH their own row to `role: "admin"`. Fixed with a `BEFORE UPDATE` trigger blocking role changes unless the caller is already admin.
- **HIGH — `store`/`media` write policies allowed any authenticated user, not just admin.** `store_auth` and the `media` bucket's delete policy checked only `authenticated`, not role. Fixed: `store_auth` now requires admin for the `superconfig` key and `can_edit_content()` (admin/editor) for everything else; `media_delete` now requires file ownership or `can_edit_content()`.
- **HIGH — task-assignee could edit any field, not just status.** `tasks_assignee`'s `WITH CHECK` didn't restrict which columns a non-admin assignee could change. Fixed with a trigger blocking title/description/assignee/due-date changes unless the caller is admin/editor.
- **HIGH — `supabase/migration.sql` had drifted from applied hotfixes.** `hotfix_chat_system_msg.sql`'s changes (anon `sender='system'` insert) are now folded into `migration.sql` directly. `hotfix_tasks_rls.sql`'s dangerous `WITH CHECK(true)` loosening was explicitly **rejected** (not folded in) in favor of the safer trigger-based fix above; the file itself is marked superseded.

Not yet done: a live smoke test confirming the new triggers/policies behave as expected against production (e.g. a non-admin test account attempting a role-escalation PATCH, or a task-assignee attempting to edit `title`, should now be rejected). Recommended before relying on these in front of a demo customer.

### Run against production, confirmed by user, 2026-07-02 (role/access remediation pass)
`supabase/hotfix_role_enforcement_2026-07-02.sql`, also folded into `supabase/migration.sql`. Run manually by the user in the Supabase Dashboard SQL Editor against `clzczbyklgdtdhgjphup`, confirmed the same day. A live smoke test (non-admin attempting to write `wsp-orgdrift`, member confirming their own `store` rows are now readable, non-admin/editor attempting a Storage upload) has not yet been separately confirmed.
- **`store_auth`**: added `wsp-orgdrift` to the admin-only key carve-out (same pattern as `superconfig`) — server-side enforcement that editor/member cannot write org-drift data, matching the UI restriction added the same session (`intranet/module-orgdrift.js`). Necessary because the whole org-drift record is one JSON blob under one `store` key — RLS cannot distinguish "create a new card" from "edit an existing card" inside it.
- **`store_read_authenticated`** (new): a broad `FOR SELECT TO authenticated USING (true)` policy. **Discovered, not originally in scope**: `store_auth` is a `FOR ALL` policy, so its USING clause governed SELECT too — with only `can_edit_content()` (admin/editor) in USING for non-`superconfig` keys, a `member` could not `SELECT` their own `store` rows at all (e.g. their own dashboard shortcuts), likely an unintended side effect of the 2026-07-01 hotfix, which was described as write-only. This policy restores read access for all authenticated users without weakening `store_auth`'s write restriction (Postgres ORs multiple permissive policies for the same command).
- **`media_insert`** (Supabase Storage): previously `WITH CHECK (bucket_id = 'media')` only — any authenticated user, including member, could upload directly via the Storage API regardless of the UI. Now requires `can_edit_content()`, matching `media_delete` (already correct).

### Run against production, confirmed by user and by direct query, 2026-07-02 (member self-created/self-edited tasks — corrected twice)
`supabase/hotfix_tasks_member_self_create_2026-07-02.sql`, also folded into `supabase/migration.sql`. Run via `npx supabase db query --linked --file ...` against `clzczbyklgdtdhgjphup` after explicit user approval — the first time this project's Supabase CLI (set up earlier the same session) was used to execute SQL rather than the Dashboard SQL Editor. Verified immediately afterward by querying `pg_policies`/`pg_proc` directly against production — all three changes confirmed present with the expected definitions. Three coordinated changes, all required together for the client UI (`intranet/module-tasks.js`) to work:
- **`tasks_self_create`** (new INSERT policy): `WITH CHECK (created_by = auth.uid() AND (assigned_to = auth.uid() OR assigned_to IS NULL))`. Lets `member` create a task for themselves (self-assigned or unassigned). Assigning a task to someone *else* still requires admin/editor (`tasks_admin`) — unaffected.
- **`tasks_assignee`** (broadened UPDATE policy): now matches `assigned_to = auth.uid() OR created_by = auth.uid()` (previously only `assigned_to`). Without this, a member-created-but-unassigned task could never be updated afterward (not even to change its own status), since the old policy only matched rows where the current user was the *assignee*.
- **`restrict_assignee_task_columns()`** (rewritten trigger): a non-admin/editor user gets full edit rights (title/description/due date/status) on a task they *created* themselves; a task *assigned to them by someone else* remains status-only (unchanged from the 2026-07-01 fix); nobody outside admin/editor can ever set `assigned_to` to a different user, regardless of who created the task.

Not yet done: a live smoke test from an actual logged-in `member` session (create a task, edit it, confirm a task assigned by someone else stays status-only) — the SQL-level verification above confirms the policies/trigger are live and match the intended logic, not that the end-to-end client flow behaves correctly against production.

Until this hotfix runs, the database still rejects a member's task-creation INSERT and any non-status edit to their own task, even though the client UI now allows both.

### Prepared, not yet run against production, 2026-07-02 (round 2 — CRM member access + tasks fully read-only)
Both SQL files below are idempotent and already folded into `supabase/migration.sql`, but per the deployment safeguard in `CLAUDE.md`/`AGENTS.md`, neither has been executed against production — awaiting explicit user approval.
- **`supabase/hotfix_crm_member_access_2026-07-02.sql`**: grants `member` read+write on exactly four `store` keys (`crm-customers`, `crm-bedrifter`, `crm-comms`, `crm-settings`) via a key-specific carve-out — not general `store` write access. Required for the CRM roles-restriction revert (see roles bullet above) to actually work end-to-end: the client UI now lets `member` create/edit customers, but until this runs the database still rejects those writes. **Caught and fixed by security-review before this was ever run**: the first draft used a single `FOR ALL` policy for the carve-out, which also covers `DELETE` — since `store` holds one JSON blob per key, that would have let `member` unconditionally delete the entire customer/company/comms/settings blob in one REST call (`DELETE FROM store WHERE key='crm-customers'`), far beyond the requested "create/edit" access. Fixed by splitting `store_auth` into command-specific policies (`store_insert_auth`, `store_update_auth`, `store_delete_auth`); `DELETE` on all keys, including the CRM ones, still requires `can_edit_content()` (admin/editor only). Both `supabase/migration.sql` and the hotfix file reflect the fixed version — neither has ever been run with the over-broad version in production.
- **`supabase/hotfix_tasks_readonly_for_assigned_2026-07-02.sql`**: narrows `tasks_assignee`'s UPDATE policy from `assigned_to = auth.uid() OR created_by = auth.uid()` to `created_by = auth.uid()` only, and simplifies `restrict_assignee_task_columns()` accordingly (the now-unreachable "status-only for assigned-by-others" branch removed). Required for the tasks fully-read-only correction to actually work end-to-end: the client now shows a read-only detail view for tasks assigned by someone else, but until this runs the database still accepts a status-only UPDATE from `member` on those rows (client-side restriction only, no server-side enforcement gap in practice since the UI no longer exposes any control to trigger it — but the RLS itself has not caught up to the new intent).

### Still open — require separate architecture work, not part of the 0.7.0 SQL batch
- **CRITICAL/HIGH — `store` table `anon` SELECT has no key restriction.** `supabase/migration.sql` (`store_anon_read` policy + `GRANT SELECT ON store TO anon`) still grants read access to every row in `store`, with no filter on which key. Since CRM customers/leads/quotes/bookings are stored in this same table (alongside legitimately-public site config), anyone holding the public anon key can read all of it directly via the Supabase REST API. `docs/architecture/storage-and-data-flow.md`'s claim that anon has no access to `store` is **verified incorrect**. Needs a real architecture fix (split public config from private customer data into separate tables/keys), not a quick policy patch. **2026-07-02 note:** the (unexecuted) `store_read_authenticated` policy prepared earlier the same day extends the same unrestricted-by-key `SELECT` to any *authenticated* role too (fixing a `member`-read regression, see 0.8.0 changelog) — it does not make this finding worse in practice (anon already had full read), but it is a second, now-authenticated-and-attributable path to the same underlying gap. **Superseded note:** `module-crm.js`'s Workspace CRM module was briefly restricted to `roles:["admin","editor"]` the same day, then **reverted** after the user clarified `member` should have normal CRM access — the app-level/route-level access path is intentionally open to `member` again as of the 0.9.0 round; it was never a fix for the underlying `store` RLS gap described here, and its removal does not change this finding's status. `member`'s CRM write access is scoped server-side via a key-specific `store_insert_auth`/`store_update_auth` carve-out (`crm-customers`/`crm-bedrifter`/`crm-comms`/`crm-settings` only, not general `store` access, and DELETE excluded) — see `docs/architecture/roles-and-tenants.md`.
- **HIGH — `chat_conversations` anon UPDATE has no visitor-ownership check (IDOR)**, and chat/visitor IDs are generated with weak entropy (`Date.now()` + 4 base36 chars, `module-chat.js`) — a guessable ID lets an anonymous client spoof another visitor's name/email or forge presence/read state on their conversation. Needs SECURITY DEFINER RPCs replacing direct anon insert/update, plus a `module-chat.js` client change — must preserve the anon visitor's ability to insert their own `sender='system'` "conversation closed" message.
- **Leads/quote/booking never reach Supabase for anonymous visitors.** `App._flushSync()` (`core.js`) only writes through to Supabase `if (!_sb || !_isAuthed)` — anonymous visitor submissions only ever land in their own browser's localStorage. Confirmed as a real bug, not a hypothesis. Needs a real table + anon-submission RPC, not an RLS tweak.
- Privacy review separately flagged: the auto-generated privacy-policy text (`computeDefaultPrivacyText()` in `core.js`) makes unverified claims (EU server location, automatic deletion) and omits Chat as a data-collection channel.
- `hub/tenants.js` plaintext password/PII — genuinely publicly deployed (`hub/index.html` exists, no deploy-exclusion mechanism, and the repo itself is public on GitHub), so a new password string would be cosmetic only. **User decision 2026-07-01: explicitly out of scope for now** — Hub is not currently used for real (non-Vibeverk-own) customers, so the actual exposure today is limited to the operator's own reference-tenant entry. Parked in `docs/roadmap/ROADMAP.md` "Ideas / Parking lot"; revisit before adding any real customer to Hub. No code change was made.
