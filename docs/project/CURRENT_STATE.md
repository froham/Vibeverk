## Implemented and verified

- **Three delivery surfaces**: public website (`/`), Workspace/intranet (`/intranet/`), Vibeverk Operator Console (`/console/`) — see `docs/architecture/system-overview.md`.
- **Public site modules**: booking, chat (widget + admin), CRM (light), FAQ, mediabank, quote/tilbud, references, scroll banner, contact/kontakt leads, user administration.
- **Workspace modules**: dashboard, announcements, tasks, notes, links, CRM, knowledge base, org drift, settings, media bank (internal), booking/quote/contact admin views. Note: several "shared" modules (`module-chat.js`, `module-crm.js`) are a single root-level file that dual-registers for both Web-admin (`App.registerModule`) and Workspace (`window.Intranet.registerModule`) — not two separate per-surface implementations. See "Known limitations" for a discovered dead-code duplicate (`intranet/module-crm.js`).
- **Roles**: `admin`/`editor`/`member` only — the database CHECK constraint on `users.role` does not permit any other value. An earlier `owner` role was removed (commit `2f8a92b`/`hotfix_roles.sql`); lingering references across code, tests and docs were cleaned up 2026-07-01, see `docs/decisions/ADR-0006-remove-owner-role-references.md`. Only `admin` can assign a task to another user (`intranet/module-tasks.js`).
- **Supabase backend**: `store` key/value table with RLS, write-through sync from `App.store`, hydration on login (steg 6a-6c). Row Level Security active on all tables; `is_admin_or_owner()` helper function in use (name is historical, checks `role = 'admin'` only).
- **Auth**: intranet and web-admin login both use Supabase Auth (email + password) whenever `config.js` has `supabase.url`/`anonKey` set — i.e. in every real customer deployment. As of 2026-07-01, `core.js`'s `renderAdminLogin()` explicitly separates "Supabase not configured" (test/local only — password fallback allowed) from "Supabase configured but the SDK failed to load" (shows a retry error, never falls back to the config password). The config-password path is now unreachable in any real, configured deployment, closing a narrow edge-case backdoor. See `docs/decisions/ADR-0003-close-admin-auth-fallback.md`.
- **Console auth**: Supabase OTP (email → 8-digit code), hardcoded superadmin email allowlist (`SUPERADMIN_EMAILS`). As of 2026-07-01, access depends solely on the allowlist + valid OTP — a leftover tenant-role check (`users.role === 'owner'`) was removed after it blocked the Vibeverk operator's own account (which had `role = 'admin'`, not `'owner'`, in the tenant's `users` table). See `docs/decisions/ADR-0004-console-access-decoupled-from-tenant-role.md`.
- **`manage-user` Edge Function**: invite/remove workspace users, service_role key never exposed to client, max 50 users/tenant enforced. Was found corrupted (truncated to 2 bytes) in the working tree on 2026-07-01 — restored from git history (`59b2dbb`) and **redeployed to production the same day**, via the Supabase Dashboard's Edge Function editor (not the CLI — see `docs/project/CHANGELOG.md` 0.3.0 "Driftsnotat" for why). The repo remains the source of truth for this function's code.
- **`send-reply` Edge Function**: outbound transactional email via Resend, supports HTML body and attachments. Verified deployed and in active use this session.
- **`features.crmFull` flag** (added 2026-07-01): governs whether direct email sending (via `send-reply`) or Outlook/`mailto:` is used for admin replies — identical behavior in Web-admin and Workspace, gated purely by customer feature flag, never by which surface the admin is using. See `docs/decisions/ADR-0002-crmfull-email-tiering.md`. As of the same day, this now also covers the CRM customer-card email dialog (previously its own unconnected mock, see Changelog 0.5.0) — `module-crm.js`/`intranet/module-crm.js` call the same shared `App.openReplyModal()` as Kontakt/Booking/Tilbud.
- **Intranet login** (`intranet/intranet-core.js`) now has the same backdoor-closure as web-admin (ADR-0005): the config-password fallback is unreachable whenever Supabase is configured, even if the SDK transiently fails to load.
- **Testing**: `node test.js` (public site) and `node test-intranet.js` (intranet) jsdom harnesses, both run in CI on every push. As of 2026-07-01 (0.7.0): 371/1 and 64/1, with the one failure in each being a known pre-existing issue (see `CLAUDE.md`/`AGENTS.md` for current test names). **Note the count changed from 372 to 371** — not a new failure, but a fix to a broken *test harness* that had been silently under-counting: `test.js` runs its bulk of assertions inside an un-awaited `(async () => {...})()` IIFE, and an earlier same-day attempt to fix a CI-hang (forcing `process.exit()` at end-of-file) exited the process before that IIFE's internal `await`s resolved — silently truncating output to ~25 of 372 tests, with no error shown, on every run since that hang-fix was added. Fixed by awaiting the IIFE's promise before exiting; full suite now completes in ~2s.
- **Email/rich-text sanitization**: `C.sanitizeRichHtml()` (`components.js`) is now applied consistently everywhere rich-text HTML is captured, including `openReplyModal`'s own editor (`core.js`) — closed a stored-XSS gap where raw `contenteditable` HTML flowed unsanitized into `send-reply` and CRM's `addComm()` history. Fixed 2026-07-01 (0.7.0).
- **CSP `connect-src`** in all four HTML entry points now uses `https://*.supabase.co`/`wss://*.supabase.co` instead of a hardcoded single-project hostname, so a fork with a different Supabase project isn't blocked. Fixed 2026-07-01 (0.7.0).

## Known limitations

- `config.js → admin.password` is still the placeholder value `"test"`. As of 2026-07-01 this is no longer reachable in production (see Auth note above), but should still be changed before any real production customer use — it remains the fallback for local/test environments without Supabase, and a placeholder value there invites confusion.
- Web-admin password authentication, when Supabase is not configured (test/local environments only), is enforced client-side only — not a production security boundary. See `docs/security/security-baseline.md`.
- `storageKey: "nordpunkt"` cannot be renamed without a full atomic data migration — existing Supabase rows and localStorage entries are keyed to it.
- `AGENTS.md` and `CLAUDE.md` are maintained as two separate, near-duplicate files (one per AI harness) rather than a single generated source — keeping them in sync is a manual, currently unenforced process.
- **`intranet/module-crm.js` is dead code.** `intranet/index.html` loads the root `module-crm.js` instead (via `../module-crm.js`), which dual-registers for both Web-admin and Workspace — the separate `intranet/module-crm.js` file is never loaded by any real page. However, `test-intranet.js` (root and `intranet/`) hardcodes evaluation of `intranet/module-crm.js` for its CRM-related assertions — meaning those tests exercise a file that never runs in production, while the Workspace-registration branch of the real, active `module-crm.js` has no dedicated Workspace-specific test coverage. Discovered 2026-07-01, not resolved — needs a decision: delete the dead file and adjust the test harness to load root `module-crm.js`, or something else.
- `is_admin_or_owner()` SQL function name is historical (only checks `role = 'admin'`) — not renamed since it's referenced by many RLS policies; would need a coordinated migration. See `docs/decisions/ADR-0006-remove-owner-role-references.md`.

## Security findings status (Fase 1 audit + Codex reconciliation, both 2026-07-01)

### Fixed in code, 2026-07-01 (0.7.0) — no Supabase change needed
- Stored XSS in `openReplyModal`'s rich-text editor (see above).
- Hardcoded CSP `connect-src` (see above).
- Missing input limits on `send-reply` Edge Function (email format, length caps, attachment count/size) — fixed in `supabase/functions/send-reply/index.ts`, **not yet redeployed to production** (code-only until redeployed via the Supabase Dashboard).
- Test harness silently truncating ~93% of `test.js`'s assertions on every run (see above) — not a security finding per se, but discovered via the same reconciliation pass and worth flagging since it means the "372/1 passing" baseline claimed in earlier sessions was **not actually being verified** for most of that period.

### Fixed in production, confirmed run by user 2026-07-01
`supabase/hotfix_security_audit_2026-07-01.sql` was run by the user in the Supabase Dashboard SQL Editor against the live project (`clzczbyklgdtdhgjphup`) — confirmed success ("Success. No rows returned"). Same fixes also folded into `supabase/migration.sql` for future fresh customer projects. Noted here as user-confirmed rather than repo-verified, since SQL execution against Supabase happens outside version control (same caveat as `hotfix_visitor_rpcs.sql` below).
- **HIGH — self-escalation to admin via `users.role`.** `users_self_update`'s `USING`/`WITH CHECK` only verified row ownership, not that `role` stayed unchanged (RLS can't restrict individual columns) — any member could PATCH their own row to `role: "admin"`. Fixed with a `BEFORE UPDATE` trigger blocking role changes unless the caller is already admin.
- **HIGH — `store`/`media` write policies allowed any authenticated user, not just admin.** `store_auth` and the `media` bucket's delete policy checked only `authenticated`, not role. Fixed: `store_auth` now requires admin for the `superconfig` key and `can_edit_content()` (admin/editor) for everything else; `media_delete` now requires file ownership or `can_edit_content()`.
- **HIGH — task-assignee could edit any field, not just status.** `tasks_assignee`'s `WITH CHECK` didn't restrict which columns a non-admin assignee could change. Fixed with a trigger blocking title/description/assignee/due-date changes unless the caller is admin/editor.
- **HIGH — `supabase/migration.sql` had drifted from applied hotfixes.** `hotfix_chat_system_msg.sql`'s changes (anon `sender='system'` insert) are now folded into `migration.sql` directly. `hotfix_tasks_rls.sql`'s dangerous `WITH CHECK(true)` loosening was explicitly **rejected** (not folded in) in favor of the safer trigger-based fix above; the file itself is marked superseded.

Not yet done: a live smoke test confirming the new triggers/policies behave as expected against production (e.g. a non-admin test account attempting a role-escalation PATCH, or a task-assignee attempting to edit `title`, should now be rejected). Recommended before relying on these in front of a demo customer.

### Still open — require separate architecture work, not part of the 0.7.0 SQL batch
- **CRITICAL/HIGH — `store` table `anon` SELECT has no key restriction.** `supabase/migration.sql` (`store_anon_read` policy + `GRANT SELECT ON store TO anon`) still grants read access to every row in `store`, with no filter on which key. Since CRM customers/leads/quotes/bookings are stored in this same table (alongside legitimately-public site config), anyone holding the public anon key can read all of it directly via the Supabase REST API. `docs/architecture/storage-and-data-flow.md`'s claim that anon has no access to `store` is **verified incorrect**. Needs a real architecture fix (split public config from private customer data into separate tables/keys), not a quick policy patch.
- **HIGH — `chat_conversations` anon UPDATE has no visitor-ownership check (IDOR)**, and chat/visitor IDs are generated with weak entropy (`Date.now()` + 4 base36 chars, `module-chat.js`) — a guessable ID lets an anonymous client spoof another visitor's name/email or forge presence/read state on their conversation. Needs SECURITY DEFINER RPCs replacing direct anon insert/update, plus a `module-chat.js` client change — must preserve the anon visitor's ability to insert their own `sender='system'` "conversation closed" message.
- **Leads/quote/booking never reach Supabase for anonymous visitors.** `App._flushSync()` (`core.js`) only writes through to Supabase `if (!_sb || !_isAuthed)` — anonymous visitor submissions only ever land in their own browser's localStorage. Confirmed as a real bug, not a hypothesis. Needs a real table + anon-submission RPC, not an RLS tweak.
- Privacy review separately flagged: the auto-generated privacy-policy text (`computeDefaultPrivacyText()` in `core.js`) makes unverified claims (EU server location, automatic deletion) and omits Chat as a data-collection channel.
- `hub/tenants.js` plaintext password/PII — genuinely publicly deployed (`hub/index.html` exists, no deploy-exclusion mechanism), so a new password string would be cosmetic only. Needs a real user decision: proper auth vs. removing Hub from public deployment.

Already fixed same day, before this pass: a BLOCKER self-escalation-to-admin path in `core.js`'s login handler (separate from the one ADR-0005 closed), and a regression in `module-users.js` that had broken the Users admin panel entirely.
